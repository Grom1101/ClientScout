# Контекст проекта: ClientScout (Telegram Mini App)

Этот документ предназначен для синхронизации всех AI-агентов (Front-end, Back-end, Security/Fix), работающих над проектом ClientScout. Пожалуйста, внимательно изучите структуру, чтобы избежать конфликтов в коде и архитектуре.

## О проекте
ClientScout — это Telegram Mini App (TMA) для автоматического поиска заказов (лидов) и рассылки откликов. Приложение собирает заказы из Telegram-чатов и фриланс-бирж (Upwork, Quark, Fiverr и др.), а также позволяет настраивать автоматические рассылки по базам чатов.

## 1. Frontend (Текущее состояние)
Фронтенд полностью сверстан по макетам и работает в режиме "dry-run" (используются мок-данные, бэкенд не подключен, инициализация Telegram SDK временно отключена для локального тестирования).

- **Стек:** React + Vite + Tailwind CSS v4 + Zustand (для стейт-менеджмента).
- **Дизайн-система:** Тёмная тема (фон #0B0E18), акцентный цвет (фиолетовый #7C3AED), скругления rounded-2xl, шрифт Inter. Отключено стандартное выделение (user-select: none).
- **Данные:** Все фейковые данные лежат в `src/data/mockData.ts`. Роутер находится в `src/App.tsx`.

### Архитектура экранов (3 основные вкладки в BottomNav):
1. **Главная (/)**
   - Дашборд: Карточки статистики (Лиды за сегодня, Рассылки).
   - График активности (Recharts).
   - Список последних рассылок.
   - Управление профилями: Приложение не использует email-регистрацию. Пользователь просто создает "внутренние профили" (например, "Frontend Russian", "Frontend English") со своими настройками. Смена и создание профиля — через модальное окно по клику на аватар в правом верхнем углу.
2. **Поиск (/search)**
   - Назначение: Поиск заказов по заданным параметрам.
   - Чаты: Открывается как отдельная страница. Список Telegram/WhatsApp чатов с группировкой по платформам.
   - Биржи: Открывается как модальное окно. Список бирж с кнопками "Подключить" (здесь будут вводиться API-ключи).
   - Настройки поиска: Открывается как модалка по клику на шестеренку ⚙️. Настройки: Периодичность парсинга (5-60 минут) и тумблер уведомлений.
   - Лента заказов: Список найденных заявок. При клике на заказ ("Написать" / "Открыть") открывается модалка с деталями заказа.
3. **Рассылка (/mailing)**
   - Назначение: Автоматическая рассылка рекламных сообщений по чатам.
   - Чаты: Отдельная страница (UI идентичен чатам поиска). Можно добавлять чаты по ссылке (платформа определяется автоматически) и удалять свайпом влево (SwipeableItem).
   - Сообщения: Модалка. Текстовое поле с поддержкой переменных и возможностью прикреплять файлы.
   - Интервал: Модалка по клику на шестеренку ⚙️. Настройка промежутка времени (круглосуточно или диапазон) и периодичности отправки.
   - Кнопки управления: Старт / Стоп рассылки.

## 2. Backend & Logic (То, что предстоит сделать)
Бэкенд-агентам нужно спроектировать API и сервисы, которые будут бесшовно интегрироваться с описанным выше UI.

### Ключевые требования к Бэкенду:
- **Система профилей:**
  - Профили привязываются к Telegram ID пользователя (т.к. это TMA).
  - Каждый профиль содержит свои настройки: список чатов, подключенные биржи, тексты сообщений, интервалы, история рассылок/лидов.
- **Интеграция с Telegram:**
  - Парсинг чатов для поиска ключевых слов (заказов).
  - Отправка сообщений в чаты от лица пользователя (userbot) или бота. Требуется грамотная архитектура во избежание банов.
- **Интеграция с биржами:**
  - Хранение API-ключей пользователей для бирж (Upwork, Quark и др.).
  - Взаимодействие с внешними API для получения потока новых заказов (периодический поллинг согласно настройкам пользователя).
- **Планировщик (Scheduler):**
  - Движок (cron/celery/bullmq), который будет управлять рассылками согласно заданным интервалам (например, каждые 30 минут с 09:00 до 23:00) и запускать поиск новых лидов.

## 3. Security & DevOps (Для агентов безопасности/ревью)
- **Безопасность API-ключей:** Ключи от бирж и сессии Telegram должны храниться в зашифрованном виде.
- **Анти-спам и Rate Limiting:** Защита от бана Telegram-аккаунтов. Имплементация задержек (jitter) при автоматических рассылках.
- **Изоляция профилей:** Строгое разграничение данных (лидов, чатов, ключей) между разными профилями и пользователями (по TG ID).
- **Валидация инпутов:** Проверка ссылок на чаты (определение платформы), защита от XSS при вводе текстов рассылки.

## Правила взаимодействия агентов
- **Для Backend:** Выдавайте JSON API, который легко "ляжет" на текущую структуру mockData.ts. Не меняйте визуальную структуру фронтенда без согласования.
- **Для Frontend:** При подключении API плавно заменяйте состояния Zustand на реальные запросы (например, React Query/SWR или axios). Сохраняйте текущие анимации модалок.
- **Для Security:** Проверяйте PR на предмет утечек токенов бота, API ключей бирж и возможных дыр в авторизации (TMA initData validation).
